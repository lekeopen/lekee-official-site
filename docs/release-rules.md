# 发布流程规则

1. **开发分支**：所有内容修改和功能开发只在 `develop` 分支进行，`main` 只用于生产发布。
2. **本地门禁**：提交或合并前在 `develop` 运行 `npm run verify`，必须全部通过。
3. **持续集成**：Pull Request 以及 `develop`、`main` 推送必须通过 GitHub Actions `quality` 检查。
4. **合并上线**：
   - 将已验证的 `develop` 合并到 `main`；
   - 推送 `main` 触发自动部署；
   - 不手工修改生产构建产物。
5. **生产冒烟**：部署后检查首页、联系页、一个新闻详情、一个项目详情、RSS，以及本次变更对应的关键资源。
6. **回滚**：生产验证失败时回退到上一条已验证的 `main` 提交并重新部署；不得在生产环境直接修补文件。
7. **发布记录**：在 `RELEASES.md` 中记录发布内容、验证结果和已知风险。
