# Site V1.2 Performance and Observability Design

## Goal

Reduce the initial JavaScript cost of the public website while preserving its current design, animation, SEO, prerendering, accessibility, content model, and deployment workflow.

## Acceptance Criteria

- Reduce the initial JavaScript entry chunk by at least 30% from the recorded V1.1 production-build baseline.
- Load page modules by route instead of including every page in the initial entry chunk.
- Achieve mobile laboratory LCP no greater than 2.5 seconds and CLS no greater than 0.1 for the homepage under the documented test profile.
- Keep every existing route prerenderable and preserve all current automated tests.
- Keep the office map, RSS, social metadata, contact workflow, and content pages working in production.

## Baseline and Measurement

The V1.1 production build emits one JavaScript entry chunk of approximately 1 MB before gzip because `src/App.tsx` imports all page modules synchronously. The build also emits a hidden source map, so total `dist/assets` size is not a useful first-load metric by itself.

V1.2 will add a deterministic build-report command that records emitted JavaScript and CSS asset sizes from `dist/assets`. The checked budget will apply to the initial entry JavaScript chunk, not source maps or route chunks. A separate mobile Lighthouse run against the production preview server will capture LCP and CLS using a documented profile; its numeric result will be recorded as release evidence rather than placed in the default CI gate, because shared GitHub runners introduce excessive timing variance.

## Loading Architecture

`src/App.tsx` will retain the existing router and route structure but load every page through `React.lazy`. A shared suspense fallback will reserve stable vertical space and provide an accessible loading status. The layout remains eagerly loaded so the header, footer, and navigation render consistently while a page chunk is fetched.

This boundary moves page-specific dependencies out of the entry chunk. In particular, `react-markdown`, `remark-gfm`, and `rehype-raw` should only load with news or project detail routes, while homepage animation code should only load with the homepage route.

Vite will own chunk naming and dependency graph decisions initially. Manual vendor chunks will only be introduced if bundle evidence shows a specific shared dependency prevents the 30% target; avoiding premature manual chunk rules reduces cache invalidation and circular-chunk risk.

## Resource and Animation Policy

Existing brand animation will remain. Motion will respect the user's reduced-motion preference, and expensive continuous effects will be reviewed for mobile cost. Images below the fold will remain lazy-loaded; above-the-fold images will receive explicit dimensions and suitable loading priority only where measurement shows a benefit.

No visual redesign or copy change is part of V1.2.

## Regression Protection

Automated tests will verify that page modules are defined with `React.lazy`, that a suspense fallback is present, and that heavy detail-page modules are not statically imported by `src/App.tsx`. A build-budget script will fail when the initial entry JavaScript exceeds the budget derived from the V1.1 baseline and the approved 30% reduction.

The canonical `npm run verify` command will build first and then run the deterministic bundle-budget check as part of the production build sequence. Existing content validation, unit tests, type checks, linting, production build, and prerendering remain mandatory.

## Delivery Sequence

1. Capture and document the V1.1 bundle and mobile performance baseline.
2. Add regression tests, route-level lazy loading, and a stable loading fallback.
3. Measure the resulting bundle and optimize only evidence-backed remaining costs.
4. Add the deterministic bundle budget and documentation.
5. Run the full local gate and mobile Lighthouse acceptance test.
6. Release through `develop`, require CI success, then fast-forward `main` and verify production.

## Rollback

The change is code-only and has no database, content-schema, or API migration. If production navigation or prerendering regresses, revert the V1.2 commits to restore synchronous page imports and remove the bundle budget. Existing static assets and content remain compatible.

## Out of Scope

- Website redesign or new business functionality.
- CMS, database, or custom API work.
- Facebook, X, or Make.com automation.
- Removing brand animation solely to improve a score.
- Playwright or permanently gating CI on timing-sensitive Lighthouse scores.
